# MailAI Portable 사용/운영 매뉴얼

## 1. 프로그램 개요

MailAI Portable는 사내 메일을 로컬 환경에서 수집하고, AI로 요약/분류/액션 추출을 수행한 뒤, 사용자가 바로 확인하고 후속 조치를 관리할 수 있게 만든 데스크톱 메일 워크스페이스입니다.

핵심 기능은 다음과 같습니다.

- 사내 IMAP 메일함에서 최근 메일 동기화
- AI 기반 메일 요약, 중요도 분류, 액션 아이템 추출
- 스레드 단위 우선순위 정리 및 후속조치 상태 표시
- 내 할 일 목록 자동 생성 및 완료 처리
- 메일 템플릿 저장, 예약 발송, 반복 발송
- 실행 로그와 발송 로그 확인
- 포터블 배포 폴더 기반 실행

현재 릴리즈 기본 AI 제공자는 사내 로컬 LLM 허브이며, OpenAI-compatible API 방식으로 연결됩니다.

## 2. 프로그램의 작동 방식

### 2.1 전체 실행 구조

프로그램은 기본적으로 다음 순서로 동작합니다.

1. `run_portable.py`가 실행됩니다.
2. 데스크톱 창은 `pywebview`로 열립니다.
3. 내부적으로 공용 서비스 컨테이너인 `AppContext`가 생성됩니다.
4. 설정, 주소록, 로그, DB, 메일 송수신, AI 분석, 스케줄러가 초기화됩니다.
5. 화면은 로컬 HTML/JS 프론트엔드가 렌더링하고, Python 쪽 동작은 `DesktopApi`가 처리합니다.
6. 앱이 켜져 있는 동안 자동 메일 동기화와 예약 발송 스케줄러가 백그라운드에서 작동합니다.

### 2.2 런타임 구성

실행 시 아래 런타임 폴더가 자동으로 준비됩니다.

- `config/`: 사용자 설정 JSON
- `data/`: SQLite 데이터베이스
- `logs/`: 앱 로그
- `cache/`: 메일 첨부 임시 캐시, 웹뷰 캐시
- `templates/`: 저장된 템플릿과 첨부파일 사본
- `prompts/`: 사용자 프롬프트 파일
- `addressbook/`: 주소록 CSV

### 2.3 실제 서비스 흐름

앱은 다음 서비스들로 구성됩니다.

- `SyncService`: IMAP에서 메일을 읽어와 로컬 DB에 저장
- `AnalysisService`: 저장된 메일을 AI로 분석
- `MailboxService`: 동기화 + AI 분석을 한 번의 사이클로 묶어서 실행
- `SendService`: 테스트 발송, 예약 발송, 반복 발송 수행
- `TemplateService`: 예약 발송용 템플릿 관리
- `MailTemplateService`: 일반 메일 작성 폼 프리셋 관리
- `SchedulerManager`: 자동 동기화와 예약 발송 스케줄 관리

## 3. 데이터가 흐르는 방식

### 3.1 메일 수집

메일 동기화 시 동작은 다음과 같습니다.

1. 설정된 IMAP 서버와 메일함에 접속합니다.
2. `sync_days` 범위 내 메일을 조회합니다.
3. 중복 `message_id`를 제외하고 새 메일만 선별합니다.
4. 메일 본문 미리보기와 첨부파일 정보를 파싱합니다.
5. 첨부파일은 `cache/mail_attachments/` 아래에 저장됩니다.
6. 메일 메타데이터와 본문 요약용 텍스트는 `data/app.db`에 저장됩니다.

### 3.2 AI 분석

AI 분석은 동기화 후 `pending` 상태 메일을 대상으로 수행됩니다.

분석 결과에는 다음 정보가 포함됩니다.

- 카테고리
- 중요도
- 한 줄 요약
- 3줄 요약
- 메일 전체 액션 아이템
- 내 액션 필요 여부
- 내 액션 항목
- 담당 주체 추정
- 액션 유형
- 기한
- 근거 문장
- 분석 사유
- 신뢰도

AI 결과는 그대로 저장하지 않고, `app/ai/analyzer.py`에서 정규화와 보정 규칙을 다시 적용합니다. 그래서 모델 출력이 조금 흔들려도 앱이 사용할 수 있는 형태로 맞춰집니다.

### 3.3 스레드 우선순위화

저장된 메일은 스레드 단위로 묶입니다. 각 스레드는 다음 정보를 기준으로 정리됩니다.

- 최신 메일 요약
- 현재 결론
- 후속조치 상태
- 기한
- 오픈 액션 개수
- 중요도/긴급도/답변 필요성/승인 필요성

UI는 이 계산을 직접 하지 않고, 이미 계산된 스레드 결과만 보여줍니다.

### 3.4 예약 발송

예약 발송은 DB에 저장된 템플릿을 APScheduler가 실행하는 방식입니다.

- `none`: 1회 발송 후 비활성화
- `daily`: 매일 반복
- `weekly`: 매주 반복
- `monthly`: 매월 반복

중요한 점:

- 예약 발송과 자동 메일 동기화는 앱이 실행 중일 때만 작동합니다.
- 창을 닫아 트레이로 숨긴 상태에서는 계속 동작할 수 있습니다.
- 앱을 완전히 종료하면 스케줄러도 중지됩니다.

## 4. 프로그램 구성

## 4.1 주요 진입점

- `run_portable.py`: 기본 데스크톱 실행 엔트리포인트
- `app/main.py`: Streamlit 호환/디버그 진입점
- `build/build_portable.py`: 포터블 빌드 및 릴리즈 폴더 생성

## 4.2 핵심 소스 폴더

- `app/ai/`: 프롬프트, AI 호출, 결과 정규화
- `app/core/`: 설정, 보안, 스케줄러, 주소록, 로거
- `app/db/`: SQLite 스키마와 저장소 로직
- `app/mail/`: IMAP/SMTP/메일 파싱
- `app/services/`: 동기화, 분석, 발송, 템플릿 관리
- `app/ui/`: 데스크톱 브리지, UI 상태 변환, Streamlit 호환 UI
- `app/ui/custom_board/`: 데스크톱 화면 HTML/CSS/JS
- `prompts/`: 기본 AI 프롬프트 텍스트
- `addressbook/`: 기본 주소록 CSV
- `tests/`: 자동 테스트

## 4.3 저장 데이터

SQLite DB인 `data/app.db`에는 다음이 저장됩니다.

- 수집된 메일
- 액션 아이템
- 예약 발송 템플릿
- 메일 작성 프리셋
- 발송 로그
- 앱 단위 동기화 상태
- 스레드 캐시

## 4.4 보안 정보 저장 위치

민감정보는 `config/settings.json`이 아니라 OS keyring에 저장됩니다.

- 메일 비밀번호
- Google Gemini API 키
- 사내 Hanlim AI API 키

즉, 설정 파일을 복사해도 키 자체는 함께 들어 있지 않습니다.

## 5. 화면별 사용 방법

## 5.1 메일 정리 대시보드

주요 용도:

- 메일 동기화 실행
- AI 분석 결과 확인
- 중요 스레드 확인
- 내 할 일 확인 및 완료 처리
- 스레드 단위 후속 메일 초안 생성

화면에서 볼 수 있는 주요 정보:

- 동기화 상태
- 자동 실행 상태
- 중요 스레드 목록
- 스레드 상세 타임라인
- 내 오픈 액션
- 완료한 액션

주요 작업:

1. `메일 동기화` 실행
2. 상단 상태 카드에서 동기화/AI 경고 확인
3. 중요 스레드 선택
4. 내 액션 항목을 확인 후 완료 체크
5. 필요 시 스레드 전체를 완료 처리
6. 필요 시 후속 메일 초안 생성

## 5.2 메일 자동발송

이 화면은 두 종류를 함께 다룹니다.

- 메일 템플릿 프리셋
- 실제 예약 발송 등록

주요 기능:

- 템플릿 새로 작성
- 발송 대상(To/Cc) 입력
- 첨부파일 선택
- 1회/매일/매주/매월 반복 설정
- 테스트 발송
- 예약 저장
- 활성/비활성 전환
- 전체 스케줄 일시중지/재개

동작 차이:

- 메일 템플릿: 작성 폼을 저장해두는 프리셋
- 발송 등록: 스케줄러가 실제로 보내는 예약 항목

첨부파일 처리 방식:

- 사용자가 고른 파일은 템플릿 전용 폴더로 복사됩니다.
- 예약 발송 시 원본 경로가 아니라 복사된 템플릿 저장본을 사용합니다.

## 5.3 발송 이력 및 로그

이 화면에서는 두 종류의 로그를 확인할 수 있습니다.

- 발송 로그
- 앱 실행 로그

발송 로그에서 확인 가능한 항목:

- 제목
- 수신자 수
- 발송 성공/실패
- 발송 시각
- 에러 메시지

앱 로그에서 확인 가능한 항목:

- 동기화 결과
- 파싱 실패
- AI 분석 실패
- 스케줄러 동작
- SMTP/IMAP 관련 예외

## 5.4 설정

설정 화면에서 관리하는 항목은 다음과 같습니다.

- 메일 주소
- 메일 비밀번호
- 이름
- 부서 / 직책
- 메일함 이름
- 동기화 보관 기간(`sync_days`)
- 배치 크기(`sync_batch_size`)
- 자동 동기화 주기(`sync_interval_minutes`)
- 스캔 상한(`sync_scan_limit`)
- 본문 미리보기 길이
- AI 타임아웃
- AI 제공자
- AI Base URL
- AI 모델
- 원문 본문 저장 여부

추가 기능:

- 메일함 목록 테스트
- 선택한 메일함 적용
- 저장된 비밀번호/API 키 삭제

현재 기본 AI 설정:

- Provider: `hanlim_openai`
- Base URL: `https://ai.hanliminve.com/llm_hub/api/v1`
- Model: `hanlimAI`

주의:

- 내부 설정 필드명은 `gemini_model`을 계속 사용하지만, Hanlim provider 선택 시에도 같은 필드에 모델명이 저장됩니다.
- 이 이름은 내부 호환성용이며 실제 호출은 Hanlim OpenAI-compatible API로 이뤄집니다.

## 5.5 도움말

도움말 화면은 간단한 빠른 시작 가이드와 문의처를 제공합니다. 실제 운영 상세는 이 문서를 기준으로 보는 것이 더 정확합니다.

## 6. 최초 사용 절차

처음 사용하는 경우 권장 순서는 다음과 같습니다.

1. 앱을 실행합니다.
2. 설정 화면에서 메일 주소와 메일 비밀번호를 입력합니다.
3. 필요하면 주소록 기반으로 이름/부서/직책이 자동 보정되는지 확인합니다.
4. 메일함 이름을 확인하고 메일함 목록 테스트를 실행합니다.
5. AI 제공자를 `사내 AI 허브`로 두고 Hanlim API 키를 입력합니다.
6. 설정을 저장합니다.
7. 대시보드로 이동해 메일 동기화를 실행합니다.
8. 중요 스레드와 내 액션 목록을 확인합니다.

## 7. 일상 사용 절차

## 7.1 메일 정리용 일반 사용 흐름

1. 앱 실행
2. 자동 동기화 또는 수동 동기화 실행
3. 대시보드에서 중요 스레드 확인
4. AI 요약과 근거 확인
5. 내 액션을 처리하고 완료 체크
6. 필요한 경우 후속 메일 초안 생성

## 7.2 정기 발송 업무 흐름

1. 자동발송 화면 이동
2. 제목/본문/수신자/첨부 설정
3. 첫 발송 일시와 발송 시간 입력
4. 반복 주기 선택
5. 테스트 발송 수행
6. 발송 등록 저장
7. 앱을 실행 상태로 유지

## 8. 주소록 사용 방법

주소록은 `addressbook/` 폴더의 최신 CSV 파일을 읽습니다.

주소록의 용도는 다음과 같습니다.

- 사용자 메일 주소 기준 이름/부서/직책 자동 보완
- 수신자 입력 시 이름으로 메일 주소 해석
- UI 자동완성 후보 제공

권장 사항:

- 최신 주소록 CSV 하나만 유지하는 것이 가장 단순합니다.
- 동일한 이름이 여러 명이면 이름만으로 수신자를 확정할 수 없습니다.

## 9. AI 프롬프트 커스터마이징

AI 프롬프트는 `prompts/` 폴더의 텍스트 파일을 읽습니다.

주요 파일:

- `classify_prompt.txt`
- `summarize_prompt.txt`
- `ownership_prompt.txt`

동작 방식:

- 실행 폴더의 `prompts/`에 사용자 파일이 있으면 그 파일을 우선 사용합니다.
- 없으면 번들된 기본 프롬프트를 사용합니다.

즉, 운영 중에도 프롬프트를 교체해 AI 출력 성향을 조정할 수 있습니다.

## 10. 설정 파일과 폴더 설명

### 10.1 `config/settings.json`

비민감 설정만 저장합니다.

- 메일 주소
- 메일함
- 동기화 주기
- AI provider
- Base URL
- 모델명
- 타임아웃

민감정보는 저장하지 않습니다.

### 10.2 `data/app.db`

앱의 핵심 로컬 저장소입니다.

### 10.3 `logs/mailai.log`

운영 이슈 추적용 로그입니다.

### 10.4 `cache/mail_attachments/`

수집된 메일 첨부 캐시입니다.

### 10.5 `templates/attachments/`

예약 발송 템플릿 첨부파일 저장 위치입니다.

### 10.6 `templates/presets/`

메일 작성 프리셋 첨부파일 저장 위치입니다.

## 11. 릴리즈 실행 방법

포터블 릴리즈는 `release/` 아래 배포 폴더로 생성됩니다.

실행 방법:

1. 릴리즈 폴더를 원하는 위치에 둡니다.
2. `MailAI_Portable.exe`를 실행합니다.
3. 최초 실행 후 `config/`, `data/`, `logs/`, `cache/`, `templates/`가 준비됩니다.

권장 사항:

- 릴리즈 폴더 구조를 임의로 변경하지 않는 것이 좋습니다.
- 예약 발송이 필요한 경우 앱을 종료하지 말고 트레이로 최소화해서 사용합니다.

## 12. 운영/개발 관점에서 알아둘 점

- 기본 사용자 실행 경로는 데스크톱 앱입니다.
- Streamlit 경로는 호환/디버그용입니다.
- 화면 데이터 구조를 바꿀 때는 `app/ui/ui_state_helpers.py`가 먼저 바뀌어야 합니다.
- UI는 우선순위 계산을 직접 하지 않고 저장소 계산 결과를 표시만 해야 합니다.
- 런타임 흐름 또는 소스 오브 트루스를 바꾸면 `docs/ARCHITECTURE.md`도 함께 갱신해야 합니다.

## 13. 문제 해결 가이드

## 13.1 메일 동기화가 안 될 때

확인 항목:

- 메일 주소 입력 여부
- 메일 비밀번호 저장 여부
- 메일함 이름이 맞는지
- IMAP 서버 접속 가능 여부
- 사내망/VPN 연결 상태

## 13.2 AI 분석이 안 될 때

확인 항목:

- Hanlim API 키 저장 여부
- AI provider가 `hanlim_openai`인지
- Base URL이 `https://ai.hanliminve.com/llm_hub/api/v1`인지
- 모델명이 `hanlimAI`인지
- 앱 로그에 HTTP/API 오류가 남는지

## 13.3 예약 발송이 안 될 때

확인 항목:

- 앱이 실행 중인지
- 예약 항목이 활성화 상태인지
- To 수신자가 비어 있지 않은지
- SMTP 로그인 가능한지
- 첨부파일이 실제로 존재하는지

## 13.4 화면에 데이터가 안 보일 때

확인 항목:

- 설정 저장 후 메일 동기화를 실제로 했는지
- AI 분석 실패 건수가 누적됐는지
- 로그 화면에서 예외를 확인했는지

## 15. 핵심 요약

이 프로그램은 단순 메일 뷰어가 아니라 다음을 함께 처리하는 업무 도구입니다.

- 메일 수집
- AI 기반 업무성 판별
- 스레드 정리
- 개인 액션 추적
- 반복 메일 발송
- 포터블 배포

실무적으로 가장 중요한 운영 포인트는 아래 세 가지입니다.

- 메일 비밀번호와 AI 키는 설정 파일이 아니라 OS keyring에 저장된다.
- 자동 동기화와 예약 발송은 앱이 실행 중일 때만 돌아간다.
- 로컬 DB와 캐시는 사용자 PC에 저장되며, `sync_days` 기준으로 오래된 로컬 메일/첨부는 정리될 수 있다.
